# Mailing Service

Сервис для импорта рассылок из XLSX-файлов с фоновой отправкой писем через Celery. Реализован как тестовое `prod-like` задание.

Принимает XLSX-файл через Django management command, валидирует каждую строку, сохраняет записи в БД батчами и ставит Celery-задачи на отправку. Повторный импорт того же файла не навредит процессу - дубликаты проверяются по `external_id` и пропускаются.

## Quick start

### Docker (рекомендуется)

Клонируем репозиторий и создаём `.env` из примера:

```bash
git clone https://github.com/Dezmound1/MailingService.git
cd MailingService
cp env.example .env
```

Поднимаем всё командами:

```bash
make build
make up
```

Будут подняты: PostgreSQL, Redis, Django (порт 8000), Celery worker.

Запуск импорта (можно указать как свой фал, так и использовать имеющийся):

```bash
make import                                                 # sample.xlsx, batch_size=500
make import DRY_RUN=1                                       # предпросмотр
make import BATCH_SIZE=100                                  # батч == 100
make import FILE=/data/other.xlsx DRY_RUN=1 BATCH_SIZE=100  # всё вместе
```

В какой-то момент понадобится почистить кэш редиса (ну а вдруг), для этого предусмотрена команда:

```bash
make redis-clean
```

Для быстрого доступа к shell (возможно появится желание удалить объекты из бд или сделать что-то ещё) приложения предусмотрена команда:

```bash
make shell
```

Остановить:

```bash
make down
```

## Задание

Реализовать механизм импорта рассылок из файла формата XLSX с последующей отправкой писем. Каждая строка файла содержит данные письма, которое должно быть отправлено пользователю. Файл может содержать большое количество записей.

### Формат файла

Первая строка - заголовки колонок:

| external_id | user_id | email | subject | message |
|---|---|---|---|---|
| ext-1 | user-1 | user@example.com | Тема | Текст письма |

Где `external_id` - уникальный идентификатор записи во внешней системе, используется для предотвращения повторной обработки при повторном импорте.

### Вывод команды

```
Import complete:
  Total rows:  1000
  Created:      950
  Skipped:       45
  Errors:         5
```

Опции: `--dry-run` для предпросмотра без записи в БД, `--batch-size` для настройки размера батча.

## Выбор стратегии

### Архитектура

Проект разбит на три слоя:

- **Model** - `MailingRecord` хранит данные рассылки и статус отправки (`pending` → `sent` / `failed`).
- **Service** - бизнес-логика. `importer.py` - парсинг, валидация, батчинг. `email.py` - Celery-задача отправки.
- **Command** - точка входа. Валидирует файл, вызывает сервис, выводит статистику.
- **Admin** - веб-интерфейс для мониторинга результатов импорта и отправки.

### Django Admin

Задание не требует UI, но хотелось иметь возможность быстро посмотреть что происходит после импорта - сколько писем ушло, какие зависли, где ошибки. Django admin решает это из коробки без написания отдельных view и шаблонов.

Модель `MailingRecord` зарегистрирована в админке с набором настроек:

- `list_display` - в таблице сразу видны `external_id`, `email`, тема, статус и дата создания. Не нужно открывать каждую запись, чтобы понять её состояние.
- `list_filter` по статусу - боковая панель с кнопками Pending / Sent / Failed. Один клик - и видишь только проблемные записи.
- `search_fields` - поиск по `external_id`, `email` и `subject`. Удобно, когда нужно найти конкретное письмо среди тысяч по любому из этих признаков.
- `readonly_fields` - сделал всё в ```readonly```, чтобы не мешать естественному процессу, да и странно редактировать логи из админки.

Суперпользователь создаётся автоматически при старте контейнера (логин/пароль из `.env`), так что после `make run` можно сразу заходить на `http://localhost:8000/admin/`.

### Батчинг

Файл может быть большим, поэтому записи обрабатываются батчами (по умолчанию 500). Каждый батч - один `SELECT` для проверки дубликатов + один `bulk_create`.

### Идемпотентность

`external_id` с `unique=True` гарантирует, что повторный импорт того же файла не создаст дубликатов. Перед `bulk_create` мы фильтруем уже существующие записи, а `ignore_conflicts=True` страхует от race condition при параллельном импорте.

### Celery

Отправка писем вынесена в Celery вместо синхронной обработки внутри команды импорта. Это даёт несколько преимуществ:

- **Параллельность** - несколько worker'ов обрабатывают письма одновременно, вместо последовательной отправки в одном процессе. Недолго думая пришел к выводу что выполнять это последовательно с задержной в 20 сек на каждой итерации оооочень долго уже при 100 записей.
- **Масштабирование** - Масштабирование отправки - это просто добавление worker'ов.

Задачи сериализуются в JSON - этого достаточно, так как передаётся только `int` pk. При исчерпании ретраев запись получает статус `failed` с текстом ошибки.

### Логирование

В задании отправка письма описана через `print "Send EMAIL..."`. Вместо этого мне захотелось использовать `structlog`. Как-то это больше похоже на ```prod-like```.

### Тесты

Использую `pytest` + `pytest-django` + `factory-boy`. Тесты работают на SQLite in-memory - не требуют PostgreSQL или Redis. Celery в тестах работает в eager-режиме (задачи выполняются синхронно). Покрыты все слои: модель, импортер, Celery-задача, management command.

```bash
make test
```

### Docstrings

Docstrings написаны в NumPy-формате (`Parameters`, `Returns` с разделителями `----------`). Однострочные - для простых методов и классов, развёрнутые - для функций с нетривиальной сигнатурой.

## Переменные окружения

| Переменная | Описание | По умолчанию |
|---|---|---|
| `POSTGRES_USER` | Пользователь БД | `postgres` |
| `POSTGRES_PASSWORD` | Пароль БД | `postgres` |
| `POSTGRES_HOST` | Хост БД | `localhost` |
| `POSTGRES_DB` | Имя БД | `mailing_service` |
| `POSTGRES_PORT` | Порт БД | `5432` |
| `REDIS_HOST` | Хост Redis | `localhost` |
| `REDIS_PORT` | Порт Redis | `6379` |
| `DJANGO_SECRET_KEY` | Секретный ключ Django | - |
| `DJANGO_DEBUG` | Режим отладки | `False` |
| `DJANGO_ALLOWED_HOSTS` | Разрешённые хосты | `localhost,127.0.0.1` |
